Scope & method

Reviewed features/roadmap.html against its shared assets docs/assets/docs.css and docs/assets/docs.js, since the page's behavior (accordions, ToC scrollspy, deep-linking) is driven by that shared script. Findings are graded:

GradeMeaning

Correctness errors fixed

The three Phase 3 sub-entries in the ToC all linked to #phase3 instead of the accordion items #p3a / #p3b / #p3c:

  • Deep-linking broke — clicking "3B · Send service" jumped to the section heading, and openHashTarget() never opened the matching accordion item, because #phase3 is the <h2>, not an .accordion-item.
  • Scrollspy highlighted the wrong link — initScrollSpy() builds a Map keyed by target element. Three links resolving to the same #phase3 target collide, so map.set(target, link) keeps only the last one. Result: only "3C · ComposeView" could ever light up; "3A"/"3B" were dead in the scrollspy.

Fix applied: point each sub-link at its real accordion id (#p3a, #p3b, #p3c). This restores both deep-linking and per-item scrollspy in one change.

Both the ToC link and the <h2 id="later"> heading read "Phases 5–9", yet the first accordion item inside is #p4 — "Reply / reply-all / forward". The section actually spans Phases 4 through 9.

Fix applied: relabeled both to "Phases 4–9".

Organization & consistency recommended

The page now declares every phase done, and Phases 4–9 were rewritten as past-tense "Done." retrospectives. But the Phase 3 section and its 3A/3B/3C accordions are still written as an unimplemented plan — imperative steps like "Teach the SMTP client…", "add a case to RequestEncoder". The "Locked scope" callout and the present-tense roadmap intro ("The order is a recommendation…") add to the mismatch.

Coverage is also lopsided: Phases 1–2 appear only as table rows, Phase 3 gets a full multi-step plan, and Phases 4–9 get one-paragraph done-notes.

Recommendation: pick one frame and apply it uniformly. Either (a) convert Phase 3 to a "Done." retrospective like 4–9 and move the step-by-step plan into a collapsed "How it was built" detail, or (b) keep the page as a historical roadmap and clearly mark every phase's status the same way. Right now a reader can't tell whether Phase 3 is shipped.

The status grid gives primary (accent) styling to a "Next phase" card whose value is an em-dash, next to "Phases remaining: 0". Emphasizing an empty/non-actionable value draws the eye to nothing.

Recommendation: with zero remaining, drop the "Next phase" card or repurpose it as a meaningful headline metric (e.g. promote "Phases done 1–9" to primary), so the accent colour marks something real.

The ToC lists three sub-entries for Phase 3 but collapses Phases 4–9 into a single "Phases 4–9" link. Now that 4–9 each have their own accordion (#p4…#p9), the navigation is asymmetric.

Recommendation: either add sub-entries for 4–9 (pointing at #p4…#p9, which deep-link correctly) or drop the Phase 3 sub-entries for symmetry. Pairs naturally with resolving O1.

DRY & single source of truth recommended

Each phase's status/title lives in (1) the status stat-cards, (2) the roadmap <table> badges, and (3) its accordion heading + body. These are hand-maintained copies — change a phase and three spots must stay in sync, which is exactly how "Phases 5–9 / contains Phase 4" (F2) crept in.

Recommendation: drive the table rows, ToC entries, and accordion shells from one small JS array of phase objects ({ id, n, title, status, touches, body }) rendered on load. This is the one place a software principle genuinely applies here — a single source of truth removes the duplication and makes label drift structurally impossible. It's optional for a doc this size, but worth it if these pages keep growing.

The page loads cdn.tailwindcss.com, but the markup uses only custom classes from docs.css (the shadcn-style CSS variables are in that file, not Tailwind's runtime) — a grep for utility classes finds none. The CDN runtime is a render-blocking fetch that compiles utilities at runtime and can cause a flash of unstyled content, for no benefit here.

Recommendation: if no sibling doc page depends on it either, drop the <script src="…tailwindcss.com"> line. Verify across the docs set before removing globally.

Accessibility & semantics optional

The .accordion-trigger buttons are real <button>s (good), but expose no aria-expanded / aria-controls, and the content panels aren't labelled. Screen-reader users get no open/closed cue. This lives in the shared docs.js, so a fix in initAccordions() (toggle aria-expanded alongside the open class) benefits every doc page, not just the roadmap.

A note on SOLID

💡 SOLID doesn't really apply to a static doc page. SOLID is about responsibilities and dependencies among classes/objects — there are none here. The principles that do apply to this artifact are DRY / single source of truth (O4), internal consistency (O1–O3), and accessible semantics (O6). I've framed the findings around those rather than forcing an OOP lens onto HTML.
