Google OAuth

Sign in to a Gmail account with OAuth 2.0 + PKCE, then read its inbox over IMAP using SASL XOAUTH2. This page documents the Phase 1 implementation — end-to-end auth that prints the ten newest inbox subjects to the console.

Overview

Gmail does not accept a password over IMAP — it requires an OAuth 2.0 access token presented through the SASL XOAUTH2 mechanism. Because the app is a native (public) client with no client secret, it uses the Authorization Code flow with PKCE (RFC 7636).

The work spans three layers, each owning one concern:

App

Thunderbird target. Drives the browser sign-in, generates PKCE, and triggers the Phase 1 print. Uses CryptoKit (Apple-only) so the shared Core stays platform-neutral.

Autoconfiguration

Core module. Builds the authorization & token URLs and performs the code-for-token exchange.

IMAP

Core module. Authenticates the IMAP connection with the bearer token via AUTHENTICATE XOAUTH2.

Sign-in sequence

From the user's tap to subjects in the Xcode console:

  1. Configure — OAuthButton resolves the provider for the email via OAuth2.request(_:) (an MX lookup matched against the canned provider configs). For Gmail this yields the google request: scope https://mail.google.com/, the bundled iOS client ID, and the reversed-client-ID redirect URI.
  2. Generate PKCE — a fresh PKCE() produces a random verifier and its challenge.
  3. Authorize — ASWebAuthenticationSession opens authURL(hint:codeChallenge:) with code_challenge + code_challenge_method=S256. The callback uses the redirect URI's custom scheme.
  4. Extract the code — the callback URL is parsed and the code query item pulled out (errors if absent).
  5. Exchange — URLSession.token(_:code:codeVerifier:) POSTs to oauth2.googleapis.com/token with the verifier and decodes access_token (and refresh_token, not yet persisted).
  6. Persist — the token becomes Token.bearer; AccountInformation assigns it to the IMAP & SMTP Server.authorization, which writes it to the keychain, and saves the account.
  7. Connect & authenticate — accessing account.imapClient connects to imap.gmail.com:993 and, because the server is .oAuth2, calls authenticateXOAuth2 instead of LOGIN.
  8. Fetch & print — MessageManager.printTopSubjects(_:) selects INBOX, fetches the highest N sequence numbers, and prints each envelope.subject newest-first.

Native clients can't keep a client secret, so PKCE binds the authorization request to the token exchange. The app sends a hash (code_challenge) at authorization time and reveals the original (code_verifier) at exchange time; only the holder of the verifier can redeem the code.

  • verifier — 32 random bytes, base64url-encoded (no padding).
  • challenge — base64url(SHA256(verifier)), sent with code_challenge_method=S256.

It lives in the app layer (CryptoKit is Apple-only) and feeds the platform-neutral Core via string parameters on authURL / tokenURL — so the Autoconfiguration module gains no crypto dependency.

struct PKCE {
    let verifier: String   // sent on the token exchange
    let challenge: String  // base64url(SHA256(verifier)), sent on the auth URL
}

URLSession.token(_:code:codeVerifier:) POSTs the authorization code to Google's token endpoint and decodes the JSON response. The client has no secret (client_secret is empty) — the PKCE code_verifier is the proof.

let urlRequest = try URLRequest.token(request, code: code, codeVerifier: codeVerifier)
let (data, response) = try await data(for: urlRequest)
guard (response as? HTTPURLResponse)?.statusCode == 200 else {
    throw URLError(.userAuthenticationRequired)
}
return try JSONDecoder().decode(TokenResponse.self, from: data).accessToken
💡Refresh token The response's refresh_token is decoded but intentionally not persisted in Phase 1 — a fresh access token is valid ~1 hour, which is plenty to read the inbox once. Persisting & refreshing is Phase 2.

Gmail authenticates IMAP via SASL XOAUTH2 with an initial response (SASL-IR), avoiding a challenge round-trip. The initial response is the raw string below; NIOIMAP base64-encodes it on the wire.

// ^A = U+0001 (Ctrl+A)
user=<email>^Aauth=Bearer <access_token>^A^A

AuthenticateCommand issues .authenticate(mechanism: AuthenticationMechanism("XOAUTH2"), initialResponse:) and reuses CapabilityHandler for the tagged result. IMAPClient.login() branches on a new IMAP.Server.Authentication flag: .oAuth2 → XOAUTH2, otherwise plain LOGIN. Account/Server.swift no longer rejects .oAuth2 for IMAP — it sets that flag and passes the stored token as the credential.

public func login() async throws {
    switch server.authentication {
    case .oAuth2:  try await authenticateXOAuth2(username: server.username ?? "", token: server.password ?? "")
    case .password: try await login(username: server.username ?? "", password: server.password ?? "")
    }
}

Known limitations

✓Resolved in Phase 2 · Milestone A Token refresh/persistence and fast XOAUTH2 failure handling have landed — see Phase 2 scope.
resolved · auth failure

A rejected token stalled until the 60s timeout. AuthenticateCommand's handler now treats the SASL error challenge / tagged NO as a typed authenticationFailed, failing fast.

resolved · token refresh

The refresh token wasn't persisted. Token now carries the refresh token + expiry (keychain), and Account refreshes before connecting and on rejection.

resolved · inbox UI

Subjects only went to print(). EmailListView now renders live IMAP messages from SwiftData; opening one fetches and displays its body (Milestones B–E).

resolved · temporary trigger

The print() trigger and the mocked TempEmail.sampleData have been removed.

Phase 2 scope

Phase 1 produced the reusable foundation; Phase 2 turns the printed subjects into a real, durable inbox. Decisions: persisted SwiftData model · list + full reading · token refresh in scope. Delivered one commit per milestone.

  1. done A · Token durability — Token carries refresh token + expiry + self-refresh coordinates (keychain); Account refreshes before connect and force-refreshes on rejection; XOAUTH2 failures fail fast.
  2. done B · Email model + SwiftData — EmailData (Sendable mapper from IMAP.Message, incl. EmailAddressProtocol → EmailAddress) + an Email @Model keyed by account + uidValidity + UID. SwiftData is app-layer; Core stays platform-neutral.
  3. done C · Inbox data source — MessageManager.fetchInbox returns the newest envelopes off-main; an @MainActor Inbox store upserts them into SwiftData with loading/error state.
  4. done D · List UI wiring — EmailListView drives off a SwiftData @Query + Inbox (load-on-appear, pull-to-refresh, empty/loading/error); EmailCellView/ReadEmailView consume Email. Mock model removed.
  5. done E · Message reading — MessageBody MIME-parses the complete message (HTML/plain, charset-aware, attachment metadata); ReadEmailView loads the body on open and marks \Seen (IMAPClient.markSeen).

Deferred to Phase 3: IDLE/push refresh, compose/send (SMTP XOAUTH2), attachment download, server-side sync/pruning, multi-account switching polish.
